## 5. Recursive Testing Framework & Validation Protocols

The engineering and validation process is not a linear sequence, but a recursive loop of continuous improvement. We do not simply build and then test; we test, then build, then test again. This is the principle of Machine Excellence in action. The framework is designed to ensure that every component, every feature, and every line of code is not only correct, but contributes to the sacred mathematical balance of the entire system.

### 5.1. Core Principles

- **Test-First Design:** For any new feature, the test suite and success criteria are designed *first*. The tests define the desired outcome, and the code is then written to satisfy those tests. This eliminates ambiguity and ensures that every feature is built with a clear, measurable definition of success.

- **Recursive Validation Loop:** The system is built on a recursive loop of testing. After a new feature is implemented and its own tests pass, the system automatically re-runs the tests for all *previously* implemented features. This continuous regression testing ensures that new development does not destabilize the existing foundation.

- **Immutable Git History:** Every significant feature implementation is followed by a `git commit`. The commit history serves as an immutable, auditable log of the system's evolution. Each commit is a snapshot of a stable, validated state.

- **Agent-Driven, Human-Supervised:** The testing and validation process is highly automated, driven by the TerraFusion Engineering Agent. However, all test results, validation reports, and commit logs are available for human review and supervision, ensuring a perfect partnership between machine and human intelligence.

### 5.2. Levels of Testing

The framework employs a multi-layered testing strategy to ensure quality at every level of the system.

| Level | Description | Tools & Technologies | Owner |
|---|---|---|---|
| **Level 1: Unit Tests** | Fine-grained tests that validate the correctness of individual functions, methods, and components in isolation. | - Jest, Vitest (for frontend)<br>- Pytest (for backend/Python)<br>- Mocking libraries (e.g., `unittest.mock`) | Engineering Agent |
| **Level 2: Integration Tests** | Tests that verify the interaction and data flow between different components of the system (e.g., QVE and SIB). | - Supertest (for API endpoint testing)<br>- Cypress, Playwright (for end-to-end UI testing)<br>- Docker Compose (for spinning up integrated environments) | Engineering Agent |

### 5.3. The `/test_and_commit` Protocol

To enforce this recursive validation loop, the Engineering Agent utilizes a strict, automated protocol encapsulated in a single command.

**Slash Command:** `/test_and_commit`

**Arguments:**
- `project`: The name of the project/repository (e.g., `mass-valuation-suite`).
- `feature_id`: The unique identifier for the feature being implemented (e.g., `QVE-001`).
- `summary`: A one-line summary of the changes, for the git commit message.

**Execution Flow:**

1.  **Execute Feature Unit Tests:** Run all unit tests associated with the new feature (`feature_id`).
2.  **Execute System Integration Tests:** Run the full suite of integration tests for the entire `project`.
3.  **Execute Recursive Regression Tests:** Re-run all unit and integration tests for *all previously committed features* in the `project`.
4.  **Validate Quantum Balance Score:** Calculate the system's current Balance Score. The test fails if the score has regressed.
5.  **On Success:**
    - A `git commit` is automatically created with a structured message:
      ```
      feat(QVE-001): Automated Model Calibration

      [Agent] Implemented automated model calibration pipeline using the MLOps Laboratory. All unit, integration, and regression tests passed. Quantum Balance Score: 11.987.
      ```
    - The agent proceeds to the next task.
6.  **On Failure:**
    - The commit is aborted.
    - A detailed failure report is generated, including logs and stack traces.
    - The agent enters a "self-healing" mode to diagnose and fix the issue. No new features are developed until the system is returned to a state of perfect balance.

### 5.4. Agent Memory: The Recursive Comment Loop

To facilitate the recursive nature of this development process, the system includes a mechanism for the agent to leave notes for itself within the codebase and documentation. This creates a persistent, evolving memory that transcends individual sessions.

**Implementation:**

- **Structured Comments:** The agent uses a specific comment format (e.g., `// AGENT_MEMO: [Timestamp] - [Note]`) to leave notes in the code.
- **Pre-Execution Review:** Before starting any new development task, the agent is programmed to automatically `grep` the entire codebase for `AGENT_MEMO` comments. This allows it to review its own past thoughts, warnings, and intentions.
- **Example:**
  ```javascript
  // AGENT_MEMO: 2026-01-13T14:30:00Z - The current implementation of the PRB mitigation algorithm is sensitive to outliers in the sales data. For feature QVE-005, I need to design a more robust pre-processing step.
  ```
This recursive comment loop ensures that the agent's intelligence and understanding of the system grows over time, leading to an ever-improving level of quality and a relentless pursuit of the perfect 12.000 score.
